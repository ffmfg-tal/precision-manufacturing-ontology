# Carbon OS Novelty Scan — 2026-02 through 2026-04

**Scope:** Carbon migrations merged after the `reference/carbon-os-mapping.md` snapshot (undated in that file; plan-of-record baseline is early-to-mid February 2026) through 2026-04-22.
**Source:** `crbnos/carbon` repo, `packages/database/supabase/migrations/` (the existing mapping's 700+ count is approximate — at scan time the live tree holds 634 migrations; all 2026 migrations were enumerated and the structurally interesting ones in the 2026-02 → 2026-04-22 window were inspected).
**Intent:** Identify novel entities/patterns worth considering in the deepening pass. This is input to that pass, not a driver of it. Carbon remains a reference example, not the canonical reference.

## Findings

### Adopt (candidates for canonical incorporation)

- **NCR system-action taxonomy (Containment / Corrective / Preventive / Verification / Communication)** — [20260313000001_issues-containment-status.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260313000001_issues-containment-status.sql) and [20260316000002_containment-status-three-state.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260316000002_containment-status-three-state.sql)
  - What Carbon added: `nonConformanceSystemActionType` enum with five canonical action classes, a `systemType` column on `nonConformanceRequiredAction`, a partial unique index so each company has at most one of each system action type, a protect-trigger that blocks deletion or re-typing of seeded system actions, and a derived `ContainmentStatus` view (`Contained` / `Uncontained` / `N/A`) driven by the presence and state of a Containment action task. A `companySettings.qualityIssueTarget` column was also added.
  - Why it matters for aerospace CM: AS9100 and customer quality manuals distinguish containment from root-cause, corrective, and verification actions. The ontology's NCR archetype and `extensions/ncr.md` currently treat corrective actions as a flat collection without a canonical action-type vocabulary. Adopting the five-value enum (and the derived containment-status rollup) gives the NCR flow first-class structure that aligns with 8D / AS13100 expectations and enables objective "is this issue contained?" reporting.
  - Where in the ontology it would land: `extensions/ncr.md` (action-type enum on corrective-action records) and `reference/entity-inventory.yaml` (new enum / classified sub-entity). The derived containment status would also be a reportable KPI in the quality archetype walk.

- **Unified ApprovalRequest / ApprovalRule workflow (document-agnostic approval gate)** — [20260119191608_approvals-workflow.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260119191608_approvals-workflow.sql) (extended by supplier-approval migrations [20260310005407_supplier-approvals.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260310005407_supplier-approvals.sql), plus subsequent `approvalDocumentType` enum additions for `supplier`, `purchaseOrder`, `qualityDocument`)
  - What Carbon added: an `approvalRequest` table keyed to `(documentType, documentId)`, an `approvalRule` table with amount thresholds / approver groups / escalation days, an `approvalStatus` enum (Pending/Approved/Rejected/Cancelled), and a `Needs Approval` state spliced into `purchaseOrderStatus`. The pattern then grew to cover supplier onboarding and quality document approval through 2026-03.
  - Why it matters for aerospace CM: aerospace CMs gate many documents — FAIR packages, quality manual revisions, supplier approvals, PO flowdowns, and NCR dispositions all need audit-worthy approval chains. The ontology currently models approvals per-entity (supplier-approval, customer-approval, FAI status) without a shared approval-request supertype. A canonical `approval_request` archetype would eliminate duplication and directly support AS9100 §7.1.5.2 / §8.3.4 / §8.7 documentation evidence.
  - Where in the ontology it would land: a new Layer-2 archetype `extensions/approvals.md` (or similar), referenced by FAI, supplier-approval, customer-approval, quality-document, and the NCR extension. The entity-matrix "cert-document supertype" taxonomic question (noted in `entity-matrix.md` line 23) could be resolved in the same shape.

- **Per-company audit log system with entity-type rollup** — [20260212152709_audit_log_system.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260212152709_audit_log_system.sql), [20260218000000_expand_audit_log_entities.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260218000000_expand_audit_log_entities.sql), [20260418000000_audit_log_add_record_id.sql](https://github.com/crbnos/carbon/blob/main/packages/database/supabase/migrations/20260418000000_audit_log_add_record_id.sql)
  - What Carbon added: a per-company `auditLog_{companyId}` table created on demand, capturing `entityType`, `entityId`, `operation` (INSERT/UPDATE/DELETE), `actorId`, `diff` (JSONB), `metadata`, and `createdAt`; an `auditLogArchive` table for cold storage; entity-type rollup so child-table changes attribute to the parent entity view; and an `AUDIT` handler-type in the event-system subscription engine (~60 tables across 21 entity types by 2026-02-18).
  - Why it matters for aerospace CM: the ontology's retention/governance layer (gap #10 in the existing Carbon mapping) currently has no concrete shape. AS9100 §7.5.3 and DFARS 252.204-7012 require tamper-evident change history on quality-relevant records. The pattern of "who changed what field when, rolled up by entity" is the right primitive; whether the ontology codifies the Carbon implementation or simply the audit-log archetype, it's worth explicit coverage.
  - Where in the ontology it would land: a new `extensions/audit-and-retention.md` archetype (or fold into a compliance extension) and a reference entry in the supplier/customer/part/job archetype walks.

## Summary

Carbon's 2026-02 → 2026-04 activity is dominated by platform hardening (event-system interceptors, audit logging, RLS refactors, search infrastructure) and transactional-ERP depth (pricing rules, approval workflows, storage nesting, method-type renames, planning functions). Aerospace-specific features are scarce — Carbon is deepening as a general-purpose manufacturing ERP, not moving toward AS9100/Nadcap coverage. The three Adopt candidates all fall in the "canonical primitives the ontology should name" category: a standard NCR-action taxonomy, a document-agnostic approval-request archetype, and an audit-log archetype for the retention/governance layer. Of these, **the NCR system-action enum is the single highest-leverage finding** — it's small, well-shaped, directly closes an existing gap in `extensions/ncr.md`, and aligns the ontology with AS13100 / 8D practice that customers will expect to see. The approval-request and audit-log patterns are bigger lifts but would resolve taxonomic open questions already flagged in `entity-matrix.md`. None of these should *drive* the deepening pass — they're well-shaped inputs for it to weigh.
